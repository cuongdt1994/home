## Important Security and Robustness Requirements

### 1. Dry-Run Mode

Add DRY_RUN=true/false in environment variables.

When DRY_RUN=true:

* The system must not execute MikroTik block commands.
* It should log:

  * “DRY-RUN: Would block IP x.x.x.x”
* It should still save AI reports and audit entries.

### 2. Whitelist Protection

Support whitelist.txt and optionally WHITELIST_CIDRS env var.

The system must never block:

* Local LAN ranges
* MikroTik router IP
* Ubuntu server IP
* Known trusted IP ranges
* Any IP/CIDR inside whitelist.txt

Even if DeepSeek says malicious, whitelist must override the AI.

Example whitelist.txt:

```text
10.100.101.0/24
10.100.101.1
10.100.101.250
192.168.0.0/16
172.16.0.0/12
```

### 3. Smart Packet / Alert Aggregation

Do not send every Suricata log line to DeepSeek.

Design an aggregation engine:

* Group by:

  * src_ip
  * event type
  * signature category
  * time window
* Example:

  * “142 SSH brute-force alerts from 185.220.101.10 against 10.100.101.250 in 5 minutes”
* Only send aggregated summaries to DeepSeek when thresholds are exceeded.
* Make thresholds configurable in thresholds.yaml.
* Prevent AI API spam and reduce cost.

Example thresholds:

```yaml
windows:
  default_seconds: 300

rules:
  ssh_bruteforce:
    min_count: 10
    window_seconds: 300
  port_scan:
    min_unique_ports: 20
    window_seconds: 300
  critical_suricata:
    min_count: 3
    window_seconds: 300
```

### 4. DeepSeek AI Orchestration

Implement flow:

```text
Suricata event
  -> normalize
  -> whitelist check
  -> aggregate
  -> threshold filter
  -> DeepSeek JSON decision
  -> validate JSON schema
  -> risk score check
  -> dry-run check
  -> MikroTik mitigation
  -> audit log

```

DeepSeek must be called with timeout.

If DeepSeek is too slow:

* Log execution time.
* Skip low-priority analysis.
* Do not block backend dashboard.
* Use asyncio queue and bounded concurrency.

### 5. DeepSeek System Prompt

Use this exact system prompt:

```text
You are a strict network security classification engine.

You will receive a summarized security event from a home/office network. Your job is to decide whether the source IP is likely malicious and whether automated blocking is justified.

Return ONLY a valid JSON object. Do not include markdown. Do not include explanations outside JSON.

Schema:
{
  "is_malicious": boolean,
  "risk_score": integer,
  "reason": string
}

Rules:
- risk_score must be an integer from 0 to 10.
- Set is_malicious=true only when the evidence strongly indicates hostile behavior such as brute force, port scanning, exploit probing, malware callback, credential stuffing, or repeated policy violations.
- Do not classify an IP as malicious from one isolated low-severity alert.
- Treat private RFC1918 IP addresses as internal unless the event clearly says they are compromised.
- If evidence is weak, ambiguous, or looks like normal LAN discovery, return is_malicious=false.
- The reason must be short, specific, and based only on the provided event summary.
- Never recommend blocking whitelisted, internal infrastructure, DNS resolvers, gateways, or monitoring systems.
```

Expected DeepSeek response:

```json
{
  "is_malicious": true,
  "risk_score": 9,
  "reason": "Repeated SSH brute-force attempts from the same external IP within 5 minutes."
}
```

Validate the response strictly before taking action.

### 6. MikroTik Firewall Strategy

Do not create one firewall rule per IP.

Use one MikroTik address-list:

```routeros
/ip/firewall/address-list
add list=ai_blacklist address=1.2.3.4 timeout=7d comment="AI Shield: reason"
```

Use one or two firewall rules only:

```routeros
/ip/firewall/filter
add chain=input src-address-list=ai_blacklist action=drop comment="AI Shield blacklist input"
add chain=forward src-address-list=ai_blacklist action=drop comment="AI Shield blacklist forward"
```

The backend should only add or refresh IPs in the address-list.

Requirements:

* Use timeout, default 7d.
* Check whitelist before SSH command.
* Deduplicate already-blocked IPs.
* Log all SSH commands and results to audit.log.
* Retry SSH failures with exponential backoff.
* Do not crash if MikroTik SSH is down.

The final project is acceptable only if:

* Dashboard looks polished, modern, and responsive.

* MikroTik health and interfaces are visible.
* MikroTik blocking uses address-list only, not one firewall rule per IP.

## Additional Requirement: Prefer Official APIs, Native Exporters, and Existing Data Sources

Do not manually reinvent low-level parsers or incomplete custom collectors when ntopng, Suricata, or MikroTik already provide official APIs, structured outputs, or native monitoring commands.

The backend should act as an integration and normalization layer, not as a fragile custom reimplementation of each product.

### 1. General Data Collection Strategy

Use this priority order for data collection:

1. Official REST API or native API.
2. Official structured JSON/log output.
3. Official CLI/SSH command with stable machine-readable output.
4. Custom parsing only as a fallback.

Avoid scraping web interfaces.

Avoid guessing undocumented fields.

Avoid relying on fragile UI HTML.

Avoid hardcoding one specific response shape without validation.

Each integration module must:

* Have its own client class.
* Use timeouts.
* Use retries with backoff.
* Validate response fields.
* Gracefully handle missing fields.
* Expose a normalized internal schema to the dashboard.
* Keep raw source response optionally for debugging.
* Never crash the backend if one external service is down.

### 2. ntopng Integration: Use Official REST API

For ntopng, use the official ntopng REST API only.

Do not scrape ntopng web pages.

The ntopng client should discover and use available REST endpoints for:

* Interfaces
* Interface statistics
* Hosts
* Host details
* Active flows
* Historical flows if available
* L7 applications
* L7 application categories
* L4 protocols
* ntopng alerts
* Traffic timeseries if available
* SNMP data if configured
* ntopng health/version/interface status if available

The implementation must not assume that all endpoints are available in every ntopng edition or version.

If an endpoint is unavailable:

* Mark that feature as unsupported.
* Show a degraded/unsupported state in the frontend.
* Continue running other features.

Create an ntopng adapter layer:

```text
ntopng REST API
  -> ntopng client
  -> response validation
  -> cache
  -> normalized dashboard schema
  -> frontend API
```

The Python backend should not try to calculate everything from raw packets if ntopng already provides the metric.

Use ntopng as the source of truth for:

* Top hosts
* Active flows
* Application protocols
* Traffic volume
* Throughput
* Host risk/alert information if available
* Interface traffic summary

Add a capability discovery function:

```text
GET /api/ntopng/capabilities
```

It should show which ntopng features/endpoints are available.

### 3. Suricata Integration: Use EVE JSON as the Official Structured Source

For Suricata, use EVE JSON as the primary source.

Do not create a custom packet parser.

Do not try to parse pcap manually unless explicitly required later.

The Suricata parser should support all common EVE event types when available:

* alert
* dns
* http
* tls
* ssh
* flow
* stats
* anomaly
* fileinfo
* netflow
* drop
* smtp
* dnp3
* smb
* rdp
* ftp
* dhcp
* quic

The parser must be schema-tolerant:

* Missing fields must not crash parsing.
* Unknown event types should be stored as raw events or counted as unsupported.
* Version differences between Suricata releases must be handled safely.
* Raw JSON should be preserved for detail view.

Use Suricata EVE fields when available:

* timestamp
* event_type
* src_ip
* src_port
* dest_ip
* dest_port
* proto
* app_proto
* flow_id
* in_iface
* alert.signature
* alert.signature_id
* alert.category
* alert.severity
* alert.action
* flow.pkts_toserver
* flow.pkts_toclient
* flow.bytes_toserver
* flow.bytes_toclient
* dns.rrname
* http.hostname
* http.url
* tls.sni
* ssh.client.software_version
* fileinfo.filename

Use `flow_id` to correlate related events.

Example correlation:

```text
alert + http + tls + flow with the same flow_id
```

The AI engine should receive aggregated summaries built from Suricata EVE data, not raw packet dumps.

### 4. MikroTik Integration: Prefer RouterOS REST API When Available, SSH Fallback

For MikroTik, support two modes:

1. RouterOS REST API mode.
2. SSH command mode fallback.

If RouterOS version supports REST API and HTTPS access is enabled, prefer REST API because it returns structured JSON-like responses and reduces fragile SSH text parsing.

If REST API is unavailable, fallback to SSH.

Add config:

```env
MIKROTIK_MODE=auto
# auto | rest | ssh

MIKROTIK_REST_BASE_URL=https://10.100.101.1/rest
MIKROTIK_REST_VERIFY_TLS=false
MIKROTIK_REST_TIMEOUT_SECONDS=8

MIKROTIK_SSH_HOST=10.100.101.1
MIKROTIK_SSH_PORT=22
MIKROTIK_SSH_USERNAME=
MIKROTIK_SSH_PASSWORD=
```

Behavior:

* `auto`: try REST API first, fallback to SSH.
* `rest`: use REST API only.
* `ssh`: use SSH only.

The MikroTik client should expose the same normalized methods regardless of backend mode:

```python
get_system_resource()
get_routerboard()
get_interfaces()
get_interface_stats()
get_firewall_address_list()
add_address_list_entry()
remove_address_list_entry()
get_firewall_rules()
ensure_ai_blacklist_rules()
```

### 5. MikroTik Data Sources to Use

Use native RouterOS resources/commands for monitoring.

Collect system health from:

```routeros
/system/resource/print

/system/package/print
```

Collect interface data from:


Collect firewall/address-list data from:

Mitigation must use address-list only:

```routeros

```

Do not create one firewall rule per blocked IP.

Only ensure these firewall rules exist once:

```routeros
/ip/firewall/filter/add chain=input src-address-list=ai_blacklist action=drop comment="AI Shield blacklist input"
/ip/firewall/filter/add chain=forward src-address-list=ai_blacklist action=drop comment="AI Shield blacklist forward"
```

### 7. Do Not Over-Code Missing Data

If ntopng, Suricata, or MikroTik does not provide a field directly:

* Do not fake it.
* Do not invent values.
* Show `unknown`, `unsupported`, or `not available`.
* Log the missing capability clearly.
* Keep the system stable.

Examples:

* If ntopng does not provide host risk score, show `N/A`.
* If MikroTik health temperature is unavailable, show `unsupported`.
* If Suricata does not emit TLS events, show TLS visibility as disabled or not seen.
* If ntopng timeseries endpoint is unavailable, fallback to backend short-term polling cache.
